Obviously, I’ve been pretty busy over the past couple of years. To start, I got a new job in late 2010 — a month after my last post — with a dating startup named HowAboutWe. It’s been an incredible experience so far that comes with many firsts:

  1. It’s my first time working with Ruby on Rails.
  2. It’s my first time using Vim regularly as an editor. (As opposed to using it quickly to change config files and such.)
  3. It’s my first time working with Git and Github.
  4. It’s my first time using compiled stylesheets via Compass and Sass.

These and a host of others came as a big shock to my system after years of working at .NET shops. Perhaps the biggest shock wasn’t simply how different this experience has been, but really how much I absolutely love the workflow. I’d been satisfactorily productive using Microsoft Visual Studio in Windows 2003 and Team Foundation Server. But getting a Github repo going, coding for a bit in Vim, committing, pushing — doing it daily here feels practically second nature to me. (Not to mention the addition of automated testing to my life.)

For a very long while now, I’ve been wanting to come back to this blog to document all of this change — and in general, get back into blogging. But I couldn’t get myself to return to it in its then-current form. To start, I desperately wanted to redesign it. Which I began to do by getting a new Wordpress theme going for my existing blog install.

But I realized quickly that the blog itself, its implementation and my use of it are as much a part of the document than the post content. I also realized that, quite frankly, if I wanted to maintain a decently regular posting schedule, then I should probably incorporate the workflows that are most productive to me.

At work, we’ve talked a lot for a while about the value of server-generated pages rather than static HTML files for certain components. For instance, we used to have ad modules that would make crazy queries to conjure seemingly timely and location-relevant 300x250s that would live on third-party websites. This was nuts for a couple of reasons — namely, that trusting user-generated content for the purpose of brand advertising is an extremely risky proposition, but also the queries were not trivial, and despite caching the hit on our servers was substantial (the ads themselves usually ran on ad-supported content sites that often see tons and tons more hits than we do, such as New York Magazine. In this case, it made complete sense to curate the content, and just serve up quick and fast static HTML that would be cached practically to infinity. A day later, we replaced nearly all of these ads with the static content, and the elegance of the solution had big payoffs in both performance, and code maintenance (a new style of ad just means a new HTML file).

Blogs were another area that we’ve talked about a lot as websites that extremely rarely change enough to necessitate being dynamically generated on the server.
